# L1 — High-Level Requirements

Project: **tab** — a personal ledger between two people for tracking loans and the counterparty's half of shared recurring bills.

This document captures **what** the system must do. The **how** (specific behaviors, validation rules, viewport breakpoints, screen-by-screen acceptance criteria) lives in `L2.md`.

---

## Functional capabilities

### L1-001: Authenticated single-user access
The system shall require a registered user to authenticate before any data can be read or written. Each authenticated user owns a private ledger that no other user can see.

### L1-002: Counterparty profile
The system shall let the authenticated user define exactly one counterparty (the person they are lending to) with a display name and optional descriptive metadata. All loans, bill splits and payments are scoped to that counterparty.

### L1-003: Loan recording
The system shall let the user record a discrete loan to the counterparty, capturing amount, date, description, method, and an optional note. Recorded loans increase the outstanding balance.

### L1-004: Recurring shared bill definitions
The system shall let the user define recurring bills (e.g. hydro, internet, phone) that the user pays in full but for which the counterparty owes a share. Each bill has a name, vendor, cadence, expected amount, due day, and split percentage.

### L1-005: Bill payment recording with automatic split
The system shall let the user mark a recurring bill as paid for a given period. When marked paid, a transaction shall automatically post the counterparty's share to the outstanding balance.

### L1-006: Payment-in recording
The system shall let the user record a payment received from the counterparty. Recorded payments-in decrease the outstanding balance and are clearly distinguished from loans in the ledger.

### L1-007: Outstanding balance calculation
The system shall maintain and display a single authoritative outstanding-balance figure for the counterparty as the sum of all loans plus all bill-split shares minus all payments-in.

### L1-008: Unified activity ledger
The system shall expose a chronological ledger of every loan, bill-split posting, and payment-in, groupable by month and filterable by type.

### L1-009: Dashboard overview
The system shall provide a dashboard that surfaces the current outstanding balance, the most recent activity, a roll-up of the current month's net change, and an anticipatory nudge for any bill due in the near future.

### L1-010: Shareable statement
The system shall produce a read-only statement that the user can share with the counterparty. The statement lists every entry contributing to the balance for a selectable period and presents totals in a neutral, factual format suitable for printing or PDF export.

### L1-011: User preferences
The system shall let the user configure currency, default bill split percentage, reminder lead time, and statement tone. Preference changes apply across the app.

### L1-012: Reminders and proactive nudges
The system shall surface reminders for upcoming bill due dates inside the app and shall not require external notification infrastructure for the MVP.

### L1-013: Data export
The system shall let the user export their complete ledger as CSV for record-keeping outside the app.

### L1-014: Sign-out and session termination
The system shall let the authenticated user explicitly end their session, invalidating their access token client-side and any associated refresh artifacts server-side.

## Cross-cutting concerns

### L1-015: Security — authentication, transport, and data protection
The system shall authenticate users via OAuth 2.0 Resource Owner Password / Authorization Code flow issuing JWT access tokens from the backend, transmit all traffic over TLS, store passwords using a modern adaptive hash (e.g. PBKDF2/Argon2), and enforce ledger isolation so no user can read or write another user's data.

### L1-016: Security — input validation and OWASP defenses
The system shall validate and sanitize every input at the API boundary, reject malformed payloads with structured errors, and defend against the OWASP Top 10 (injection, broken access control, XSS, CSRF where relevant, SSRF, etc.).

### L1-017: Performance and responsiveness
The system shall return interactive API responses within 300 ms at the 95th percentile under expected single-user load, render the first meaningful paint of any page within 1.5 s on a mid-range mobile device on a 4G connection, and handle ledgers of at least 10 000 entries without UI degradation.

### L1-018: Responsive web UI across XS–XL viewports
The system's web UI shall be a single responsive application that adapts gracefully across the standard breakpoint set — XS (< 576 px), S (≥ 576 px), M (≥ 768 px), L (≥ 992 px), XL (≥ 1200 px) — using a mobile-first approach with no horizontal scrolling at any width.

### L1-019: Accessibility
The system shall meet WCAG 2.1 AA for color contrast, keyboard navigation, focus visibility, semantic structure, and screen-reader labeling on every screen.

### L1-020: Observability
The system shall emit structured logs for every API request, every authentication event, and every domain mutation, with correlation IDs that link a frontend action to its backend trace.

### L1-021: Frontend architecture — Angular standards
The system's frontend shall be an Angular application using signals for reactive state (preferring signals over RxJS where both are viable), no NgRx, the BEM naming convention for component CSS, one TypeScript type per file with no file containing more than one exported type, and no single-file components (template and styles live in their own files alongside the component class).

### L1-022: Frontend architecture — three-library workspace
The system's frontend shall be structured as an Angular workspace containing three buildable libraries plus one application: an **api** library exposing service interfaces and DTO models plus injection tokens for those interfaces, a **components** library containing every reusable UI component, and a **tab** application that consumes both libraries and provides only the shell — routing, route guards, pages composing components, and bootstrap.

### L1-023: Frontend architecture — interface-first public API
The system's api library shall expose its public surface as TypeScript interfaces with companion `InjectionToken<T>` exports, so that consumers always depend on interfaces and never on concrete implementations. Concrete service implementations shall live behind those tokens and shall be the only place HTTP calls are made.

### L1-024: Backend architecture — .NET, controllers, MediatR, EF (no repositories)
The system's backend shall be a .NET Web API using thin controllers that dispatch to MediatR (free version) request handlers, Entity Framework Core directly inside those handlers (no repository pattern), and shall expose its DbContext to handlers as an `ITabDbContext` interface (not the concrete class) so handlers are testable in isolation.

### L1-025: Backend architecture — persistence
The system shall persist data in SQL Server (Express edition or LocalDB acceptable for development) accessed via Entity Framework Core with code-first migrations.

### L1-026: Backend architecture — one type per file
The system's backend shall follow a strict "one public type per file" rule. Every command, query, handler, response, validator, controller, entity, configuration class, and DTO lives in its own `.cs` file named for the type it contains.

### L1-027: API contract stability and versioning
The system's HTTP API shall be versioned via a URL prefix (`/api/v1/...`) and shall return structured, machine-readable error payloads conforming to RFC 7807 (Problem Details for HTTP APIs).

### L1-028: Maintainability and tooling
The system shall ship with build, test, lint, and format commands runnable from a single root command, with CI-ready scripts. Acceptance tests shall be the canonical source of truth that a feature is "done", and every acceptance test file shall declare which L2 requirement(s) it covers in a header comment.
