# L2 — Detailed Requirements

Every L2 traces to exactly one L1 in `L1.md`. Acceptance criteria are written Given/When/Then so they map directly to ATDD test cases. Each acceptance test file written by the team must declare its covered L2(s) in a header comment:

```typescript
// Acceptance Test
// Traces to: L2-014, L2-018
// Description: Counterparty bill auto-split posts to ledger
```

The viewport-breakpoint shorthand used below is fixed:
**XS** < 576 px, **S** ≥ 576 px, **M** ≥ 768 px, **L** ≥ 992 px, **XL** ≥ 1200 px.

---

## Authentication & session — traces to L1-001 / L1-015

### L2-001: Sign-in screen
**Traces to:** L1-001

The application shall present an unauthenticated user with a sign-in screen offering email + passcode entry and a "magic link" fallback. The screen shall be reachable at `/login`.

**Acceptance Criteria:**
1. Given an unauthenticated user navigates to any protected route, when the route loads, then the user shall be redirected to `/login` with the original route preserved as a `returnUrl` query parameter.
2. Given a user is on `/login`, when the page renders at XS width, then the form shall occupy the full viewport width with a 16 px minimum gutter and no horizontal scroll.
3. Given a user is on `/login`, when the page renders at XL width, then the form shall be centered as a framed card no wider than 480 px.
4. Given a user submits valid credentials, when the backend returns a JWT, then the token shall be stored in memory and a refresh artifact in an httpOnly secure SameSite=strict cookie, and the user shall be navigated to the `returnUrl` or `/dashboard`.
5. Given a user submits invalid credentials, when the backend responds 401, then the form shall display a non-revealing error ("Email or passcode is incorrect") without indicating which field was wrong.

### L2-002: JWT issuance from backend (OAuth 2.0)
**Traces to:** L1-015

The backend shall expose an OAuth 2.0 token endpoint at `POST /api/v1/oauth/token` that accepts a Resource Owner Password Credentials grant for first-party clients and returns a signed JWT access token plus a refresh token.

**Acceptance Criteria:**
1. Given a valid `grant_type=password` request with correct email and passcode, when the endpoint is invoked, then the response shall be `200` with `{ access_token, token_type: "Bearer", expires_in, refresh_token }`.
2. Given the access token is decoded, when its claims are inspected, then it shall contain `sub` (user id), `iss`, `aud`, `exp`, `iat`, and shall be signed with RS256 using a key held only by the backend.
3. Given an access token has expired, when any protected endpoint is called with it, then the API shall return `401` with `WWW-Authenticate: Bearer error="invalid_token"`.
4. Given a refresh token request, when invoked at `POST /api/v1/oauth/token` with `grant_type=refresh_token`, then a new access token and refresh token shall be issued and the previous refresh token shall be invalidated (rotation).
5. Given a passcode is stored, when it is persisted, then it shall be stored as an Argon2id or PBKDF2-SHA256 hash with a per-user salt and never in plaintext.

### L2-003: Route guards on protected routes
**Traces to:** L1-001

Every page other than `/login` shall be protected by an Angular `CanActivate` guard that verifies the presence and validity (not expired) of an access token.

**Acceptance Criteria:**
1. Given an unauthenticated user navigates to `/dashboard`, when the guard executes, then the user shall be redirected to `/login?returnUrl=%2Fdashboard`.
2. Given an authenticated user with an expired token navigates to `/loans`, when the guard executes, then the app shall attempt a silent refresh, and on success continue to `/loans`, on failure redirect to `/login`.
3. Given a user signs out, when they navigate to any protected route afterwards, then the guard shall redirect them to `/login`.

### L2-004: Sign-out
**Traces to:** L1-014

The application shall let an authenticated user sign out from the Settings page.

**Acceptance Criteria:**
1. Given a user is on `/settings`, when they tap "Sign out", then the access token shall be discarded from memory, the refresh cookie shall be cleared, the backend shall invalidate the refresh token server-side, and the user shall be navigated to `/login`.
2. Given a user has signed out, when they press the browser back button, then they shall not see any cached protected page content.

### L2-005: Per-user ledger isolation
**Traces to:** L1-015

The backend shall reject any request that would read or write a ledger entry belonging to a user other than the caller (`sub` claim).

**Acceptance Criteria:**
1. Given user A is authenticated, when user A requests `GET /api/v1/loans/{id}` for a loan owned by user B, then the response shall be `404` (not `403`, to avoid leaking existence).
2. Given user A is authenticated, when user A submits any write operation with a payload referencing user B's counterparty, then the request shall be rejected with `404` on the referenced resource.

## Counterparty — traces to L1-002

### L2-006: Single counterparty per user
**Traces to:** L1-002

Each user shall have at most one counterparty record. The counterparty shall be created automatically on first sign-up with a default placeholder name and editable from Settings.

**Acceptance Criteria:**
1. Given a user has just signed up, when they reach the dashboard, then a counterparty exists with name "Counterparty" and no other metadata.
2. Given a user is on `/settings`, when they edit the counterparty name to "Raymond Brown", then the change shall persist and the new name shall appear on `/dashboard`, `/loans`, `/statement`, and in the avatar initials throughout the app.
3. Given the counterparty name is empty, when the user attempts to save, then validation shall fail with "Name is required".

## Loans — traces to L1-003

### L2-007: Add a loan entry
**Traces to:** L1-003

The application shall let the user record a loan via a dedicated "Add a loan" dialog opened from the dashboard or from `/loans`, capturing amount (required, > 0), description (required), date (required, defaults to today, not in the future), method (optional free text), and note (optional, ≤ 280 chars).

**Acceptance Criteria:**
1. Given the user activates "Add a loan" from the dashboard or `/loans`, when the dialog opens, then its title shall read "Add a loan", the amount label shall read "Amount lent", the save action shall read "Save loan", no entry-type tablist shall be present, and the amount input shall auto-focus.
2. Given the amount input is empty or 0, when the user taps "Save loan", then the save button shall be disabled and a "Enter an amount greater than zero" hint shall be shown.
3. Given a user enters amount 120.00, description "Groceries", date 2026-05-18, method "Cash", and taps "Save loan", then the loan shall be persisted with those exact values and the dialog shall close.
4. Given a user enters a date in the future, when they attempt to save, then validation shall fail with "Date cannot be in the future".
5. Given a loan is saved, when the dashboard reloads, then the new entry shall appear at the top of Recent Activity and the outstanding balance shall increase by the entered amount.

### L2-008: Loans list grouped by month
**Traces to:** L1-008

The `/loans` page shall list every transaction (loans, bill splits, payments-in) in reverse-chronological order grouped by calendar month.

**Acceptance Criteria:**
1. Given the user has activity across three months, when they open `/loans`, then they shall see three month group headers in reverse-chronological order, each showing the net change for that month with a leading `+` or `−`.
2. Given the user filters by "Loans", when the filter is selected, then only loan-type entries shall be visible and the summary strip totals shall not change.
3. Given the user has no activity at all, when they open `/loans`, then a friendly empty state shall be shown reading "No entries yet" with a primary action linking to `/add?mode=loan`.
4. Given the viewport is L width, when the user opens `/loans`, then month groups shall flow into a 2-column responsive grid; at XS/S/M they shall stack vertically.

### L2-009: Edit a loan entry
**Traces to:** L1-003

The application shall let the user open and edit any loan entry created by them.

**Acceptance Criteria:**
1. Given a loan appears in any list, when the user activates that row, then they shall navigate to an editor pre-populated with the loan's current values.
2. Given the user changes the amount and saves, when the save returns successfully, then the new amount shall be reflected in every place the entry appears, and the outstanding balance shall recalculate.

### L2-010: Delete a loan entry
**Traces to:** L1-003

The user shall be able to delete a loan entry from the editor with explicit confirmation.

**Acceptance Criteria:**
1. Given the user is editing a loan, when they tap "Delete", then a confirm prompt shall appear ("Delete this entry? It will be removed from the ledger and statement.").
2. Given the user confirms deletion, when the operation succeeds, then the entry shall be removed from all lists and the outstanding balance shall decrease by the loan amount.
3. Given the entry was already used in a statement that has been shared, when deletion is attempted, then a warning copy shall be shown but the operation shall still proceed.

## Recurring bills — traces to L1-004 / L1-005

### L2-011: Define a recurring bill
**Traces to:** L1-004

The user shall be able to define a recurring bill with name, vendor (optional), cadence (monthly only for MVP), expected amount, due day-of-month (1–28), and split percentage (default per user preference, override allowed).

**Acceptance Criteria:**
1. Given the user is on `/bills` and taps "+ New", when the editor opens, then it shall capture all of: name (required), vendor (optional), expected amount (required, > 0), due day (required, 1–28), split percentage (required, 1–99, defaults to user preference).
2. Given the user saves a valid bill definition, when it persists, then it shall appear in the bill list with the next-due date computed for the current or next calendar month.
3. Given the user enters split percentage 100, when they attempt to save, then validation shall fail with "Split must be between 1 and 99 percent".

### L2-012: Bill list and next-due display
**Traces to:** L1-004

The `/bills` page shall list every bill definition as a card showing name, vendor, expected amount, counterparty's share, and a badge indicating the next due date or days remaining.

**Acceptance Criteria:**
1. Given a bill is due in fewer than 7 days, when it renders in the list, then its badge shall read "Due in Nd" with a leading status dot.
2. Given a bill is due in more than 7 days, when it renders in the list, then its badge shall read "Next: MMM DD" in a quiet style.
3. Given the viewport is XS/S, when `/bills` renders, then bills shall stack in a single column.
4. Given the viewport is M, when `/bills` renders, then bills shall flow in a 2-column grid.
5. Given the viewport is XL, when `/bills` renders, then bills shall flow in a 3-column grid.

### L2-013: Mark bill paid in full → automatic counterparty posting
**Traces to:** L1-005

When the user marks a bill as paid in full for a given period, the system shall create a ledger entry of type "bill-split" with amount = `expected_amount × split_percentage`, dated on the payment date, referencing the bill definition and period.

**Acceptance Criteria:**
1. Given the user taps "Mark paid in full" on the Hydro card (expected $168, split 50%), when the action confirms, then a ledger entry shall be created with amount $84.00, type "bill-split", and the outstanding balance shall increase by $84.00.
2. Given the same bill is already marked paid for the current period, when the user taps "Mark paid in full" again, then a "Period already recorded" message shall be shown and no duplicate entry shall be created.
3. Given the actual paid amount differs from the expected amount, when the user uses "Log this month" instead and enters $156.84, then the counterparty's share shall be $78.42 and the original expected amount on the definition shall not change.
4. Given the user activates "Log this month" on a bill card, when the dialog opens, then its title shall read "Log a bill payment", the amount label shall read "Amount paid", the save action shall read "Save bill payment", the form shall capture amount, description, date, and method (no note field), and no entry-type tablist shall be present.

### L2-014: Edit a recurring bill definition
**Traces to:** L1-004

The user shall be able to edit a bill definition. Edits shall apply only to future postings; previously posted bill-split entries shall not be retroactively altered.

**Acceptance Criteria:**
1. Given a bill posting from April 2026 exists at $40.50, when the user edits the bill split percentage from 50% to 60%, then the April entry shall remain $40.50 and only future markings shall use 60%.
2. Given the user changes the expected amount, when a future "Mark paid in full" is performed, then the new expected amount shall be used.

### L2-015: Delete a recurring bill definition
**Traces to:** L1-004

The user shall be able to archive a bill definition. Archived bills shall not generate new postings but historical postings shall remain in the ledger.

**Acceptance Criteria:**
1. Given a bill has 12 historical postings and the user archives it, when the user returns to `/loans`, then those 12 entries shall still appear in the ledger.
2. Given a bill is archived, when the user opens `/bills`, then the bill shall no longer appear in the active list and "Mark paid" and "Log this month" actions shall be unavailable.

## Payments in — traces to L1-006

### L2-016: Record a payment received
**Traces to:** L1-006

The user shall be able to record a payment received from the counterparty via a dedicated "Record a payment" dialog opened from the dashboard or `/loans`, capturing amount (> 0), date (default today), and method (optional).

**Acceptance Criteria:**
1. Given the user activates "Record a payment", when the dialog opens, then its title shall read "Record a payment", the amount label shall read "Amount received", the save action shall read "Record payment", the form shall capture only amount, date, and method (no description and no note), and no entry-type tablist shall be present.
2. Given the user records a payment-in of $100 on 2026-04-14, when it saves, then the outstanding balance shall decrease by $100 and a ledger entry of type "payment-in" shall appear with a leading "−" sign and muted styling on every list it appears in.
3. Given the user records a payment-in larger than the outstanding balance, when it saves, then the balance shall display the resulting negative value (overpayment) without error, since the user may want to track future advances.

## Balance & dashboard — traces to L1-007 / L1-009

### L2-017: Outstanding balance computation
**Traces to:** L1-007

The outstanding balance for the counterparty shall equal `sum(loans) + sum(bill-split postings) − sum(payments-in)` evaluated server-side.

**Acceptance Criteria:**
1. Given loans totaling $505, bill-split postings totaling $879.50, and payments-in totaling $100, when the balance is requested via `GET /api/v1/balance`, then the response shall be exactly `1284.50`.
2. Given a new entry is created via any path, when the next GET to `/api/v1/balance` occurs, then the new value shall reflect the change with no client-side calculation needed.

### L2-018: Dashboard layout & content
**Traces to:** L1-009

The `/dashboard` page shall present, in this order: a hero card containing the outstanding balance and quick actions ("Add a loan", "Log a bill"), a nudge for the nearest upcoming bill due, a recent-activity list (up to 6 most recent entries), and a current-month summary (lent, bills, paid back, net change).

**Acceptance Criteria:**
1. Given the dashboard loads, when rendered, then the hero card shall show the outstanding balance with size scaling fluidly from 48 px at XS to 88 px at XL.
2. Given there is at least one bill due within the user's reminder lead time, when the dashboard renders, then a dashed-border nudge shall appear naming the bill, the due-in-N-days, and the counterparty's expected share.
3. Given there are zero bills due within the reminder window, when the dashboard renders, then the nudge area shall not be displayed at all (no empty placeholder).
4. Given viewport M or larger, when the dashboard renders, then recent activity and the monthly summary shall sit side-by-side in a two-column grid; at XS/S they shall stack vertically.
5. Given the user has no activity, when the dashboard loads, then the hero shall show "$0.00", recent activity shall show an empty state, and the monthly summary shall show all zeros.

## Statement — traces to L1-010

### L2-019: Generate a statement
**Traces to:** L1-010

The user shall be able to view a read-only statement at `/statement` showing every contributing entry for a selectable period, with rolling totals and a final balance owing.

**Acceptance Criteria:**
1. Given the user opens `/statement`, when the page loads, then the default period shall be from the first day of the earliest entry's month to today, and entries shall be listed in chronological order with date, description, type label, total amount (where applicable), and counterparty share.
2. Given the period contains a payment-in, when the statement renders, then the payment row shall display its amount with a leading "−" and muted styling.
3. Given the statement totals are shown, when summed, then "Loans + Bills − Payments = Balance owing" and the final figure shall match `GET /api/v1/balance` for the same period.
4. Given the user prints the statement (Ctrl/Cmd-P), when the print stylesheet applies, then the top header bar and the action bar shall be hidden, and the page shall fit cleanly on A4/Letter without horizontal overflow.

### L2-020: Share statement as link
**Traces to:** L1-010

The user shall be able to generate a unique, expiring, read-only public URL for a statement that requires no login to view.

**Acceptance Criteria:**
1. Given the user taps "Send to Ray" / "Share", when the action confirms, then the backend shall mint a one-time signed URL valid for 14 days, scoped to that statement's period, and copy it to the clipboard.
2. Given an expired share link is opened, when the page loads, then the viewer shall see "This statement is no longer available" and no entries shall be visible.
3. Given a share link is opened, when the page loads, then the viewer shall see the statement read-only with no editing affordances, no navigation to other pages, and no PII beyond what the statement contains.

## Security — traces to L1-015 / L1-016

### L2-023: Input validation at the API boundary
**Traces to:** L1-016

Every write endpoint shall validate incoming payloads with FluentValidation and reject malformed input with a `400` Problem Details response.

**Acceptance Criteria:**
1. Given a loan creation request with a negative amount, when the handler is invoked, then the response shall be `400` with a Problem Details body listing `amount` as the offending field and "must be greater than 0" as the message.
2. Given a loan creation request with a `description` longer than 280 characters, when invoked, then the response shall be `400` with `description` flagged.
3. Given any request body fails JSON parsing, when received, then the response shall be `400` with `application/problem+json`.

### L2-024: TLS, security headers, and CSRF posture
**Traces to:** L1-015 / L1-016

The application shall be served only over HTTPS in non-development environments, shall set strict security headers, and shall be immune to CSRF on its JSON API.

**Acceptance Criteria:**
1. Given a request arrives over plain HTTP in production, when received, then the server shall respond `301` to the HTTPS equivalent.
2. Given any HTTP response from the backend, when inspected, then it shall include `Strict-Transport-Security: max-age=31536000; includeSubDomains`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, and a `Content-Security-Policy` permitting only the app's own origin and `'self'` for scripts and styles.
3. Given a JSON API request arrives without the `Authorization: Bearer` header on a protected endpoint, when received, then it shall be rejected `401` regardless of cookies (no implicit cookie auth).

### L2-025: Rate limiting on auth endpoints
**Traces to:** L1-016

The OAuth token endpoint shall be rate limited per IP and per account.

**Acceptance Criteria:**
1. Given five failed sign-in attempts for the same account within five minutes, when a sixth attempt is made, then the response shall be `429` with `Retry-After` set, regardless of credential correctness.
2. Given a sixth attempt within five minutes from the same IP across different accounts, when made, then the response shall be `429`.

## Performance — traces to L1-017

### L2-026: API response time budget
**Traces to:** L1-017

Every read endpoint shall return within 300 ms at the 95th percentile under a single-user load on a ledger of 10 000 entries.

**Acceptance Criteria:**
1. Given a seeded test database with 10 000 ledger entries for one user, when `GET /api/v1/loans?month=2026-05` is invoked 100 times sequentially, then the 95th-percentile response time shall be ≤ 300 ms on the CI hardware.
2. Given the dashboard endpoint returns balance + recent activity + monthly summary as a single composite call, when invoked, then it shall produce ≤ 1 SQL round-trip per logical query (balance, recent, summary), no N+1 patterns.

### L2-027: Frontend first meaningful paint budget
**Traces to:** L1-017

The dashboard shall render its hero balance within 1.5 s of navigation on a throttled 4G mid-range mobile profile.

**Acceptance Criteria:**
1. Given a Lighthouse mobile audit with 4G throttling, when `/dashboard` is measured, then First Contentful Paint shall be ≤ 1.5 s and the largest contentful element (hero balance) shall be ≤ 2.0 s.
2. Given the app bundle is built for production, when measured, then the initial route's JavaScript payload (parsed + compressed) shall be ≤ 200 KB gzipped.

## Responsive design — traces to L1-018

### L2-028: Mobile-first layout shell
**Traces to:** L1-018

The app shell shall use a mobile-first layout with breakpoints at S 576 / M 768 / L 992 / XL 1200.

**Acceptance Criteria:**
1. Given any page renders at XS width, when inspected, then no element shall cause horizontal scrolling, all tap targets shall be ≥ 44 × 44 px, and the bottom navigation bar shall be visible.
2. Given any app page renders at L or XL width, when inspected, then the bottom navigation shall be replaced by a 240 px vertical sidebar on the left, and the content area shall be capped at 1200 px wide centered horizontally.
3. Given a user resizes the browser across breakpoints, when crossing each breakpoint, then layout changes shall be triggered by CSS media queries only (no JavaScript-driven layout switching).

### L2-029: Fluid typography and spacing
**Traces to:** L1-018

Type sizes and spacing scales shall use `clamp()` to scale fluidly between breakpoints.

**Acceptance Criteria:**
1. Given the hero balance is rendered at the smallest supported width (320 px), when measured, then its font size shall be no smaller than 48 px.
2. Given the hero balance is rendered at XL, when measured, then its font size shall be no larger than 88 px.
3. Given the viewport is between two breakpoints, when the page is examined, then no font size shall "step" abruptly — values shall interpolate smoothly.

### L2-030: Bill grid responsiveness
**Traces to:** L1-018

The `/bills` page bill-card grid shall be 1 column at XS/S, 2 columns at M/L, and 3 columns at XL.

**Acceptance Criteria:**
1. Given six bills, when the page renders at XS, then a single column of six cards shall be shown.
2. Given six bills, when the page renders at M, then a 2 × 3 grid shall be shown.
3. Given six bills, when the page renders at XL, then a 3 × 2 grid shall be shown.

## Accessibility — traces to L1-019

### L2-031: Keyboard navigation
**Traces to:** L1-019

Every interactive control shall be reachable, operable, and visibly focused with the keyboard.

**Acceptance Criteria:**
1. Given a user uses only the Tab and Shift+Tab keys, when they traverse `/dashboard`, then focus order shall match visual reading order and every interactive element shall be reachable.
2. Given any focused element, when inspected, then a 2 px outline of color `rgba(180,180,188,0.55)` with 2 px offset shall be visible.
3. Given a modal or sheet is opened, when it appears, then focus shall move into it and be trapped within it until it closes, and on close shall return to the element that opened it.

### L2-032: Color contrast
**Traces to:** L1-019

Every text-on-background pairing in the UI shall meet WCAG 2.1 AA contrast.

**Acceptance Criteria:**
1. Given primary text (`#B4B4BC` / `#E6E6EA`) on the base background (`#2E2E30`), when measured, then the contrast ratio shall be ≥ 4.5 : 1 for normal text and ≥ 3 : 1 for ≥ 18 px text.
2. Given muted text (`#77777C`) is used, when audited, then it shall appear only on text ≥ 18 px or 14 px bold (large text rule).

### L2-033: Semantic structure and screen-reader labels
**Traces to:** L1-019

Every component shall use semantic HTML or appropriate ARIA roles.

**Acceptance Criteria:**
1. Given the bottom navigation, when read by a screen reader, then it shall be announced as a navigation landmark with each tab announced as its label (Home / Loans / Bills / Settings) and the active tab announced as "current".
2. Given the hero balance, when read by a screen reader, then it shall be announced as "Uncle Ray owes one thousand two hundred eighty-four dollars and fifty cents" (or equivalent), not character-by-character.

## Observability — traces to L1-020

### L2-034: Structured logging with correlation IDs
**Traces to:** L1-020

Every HTTP response shall carry a correlation ID header, and the same ID shall appear in every log line emitted during that request's processing.

**Acceptance Criteria:**
1. Given a frontend HTTP request is made, when sent, then it shall include an `X-Correlation-Id` header with a fresh UUIDv4 if none was supplied.
2. Given a backend processes a request with `X-Correlation-Id`, when logs are emitted, then every log line shall include that ID as a structured field.
3. Given an exception is thrown in a handler, when caught by the global error filter, then the response shall be a Problem Details body that includes the correlation ID under a `traceId` field, and the stack trace shall be in the server log but not in the response.

## Frontend architecture — traces to L1-021 / L1-022 / L1-023

### L2-035: Angular workspace structure
**Traces to:** L1-022

The frontend shall be an Angular workspace containing exactly: `projects/api` (library), `projects/components` (library), and `projects/tab` (application).

**Acceptance Criteria:**
1. Given the workspace, when `ng build api`, `ng build components`, and `ng build tab` are each run, then each shall succeed independently.
2. Given the `tab` application's `package.json` / `tsconfig` paths, when inspected, then it shall import from `@tab/api` and `@tab/components` via path mappings, never via relative paths into the libraries' sources.
3. Given the `components` library, when audited, then it shall not import anything from `projects/tab`.
4. Given the `api` library, when audited, then it shall not import anything from `projects/components` or `projects/tab`.

### L2-036: Interface-first public API with injection tokens
**Traces to:** L1-023

Every service in the `api` library shall be exposed via a TypeScript interface, with an `InjectionToken<T>` of the same name, and a default concrete implementation provided in a `providers` factory.

**Acceptance Criteria:**
1. Given a consumer wants to call the loans service, when they inject it, then they shall inject by token (e.g. `inject(LOANS_SERVICE)`) typed as the interface `ILoansService`, never as the concrete class.
2. Given the `api` library's public `index.ts`, when inspected, then the only exports for each service shall be: the interface, the injection token, the DTO/model types, and the `provide…` factory function. The concrete class shall not be in the public surface.
3. Given a test wants to substitute a fake loans service, when it configures the test bed, then `provide(LOANS_SERVICE, { useValue: fake })` shall be sufficient with no other knowledge of the real implementation.

### L2-037: One type per file
**Traces to:** L1-021

No `.ts` file in any frontend project shall export more than one type. Component templates and styles shall be in separate files (no inline templates, no inline styles).

**Acceptance Criteria:**
1. Given any file in the workspace, when its exports are counted, then it shall have at most one default or named exported class/interface/type/enum.
2. Given any `@Component` decorator, when inspected, then it shall use `templateUrl` and `styleUrls`, never `template` or `styles`.
3. Given a model type and its companion interface, when authored, then they shall live in separate `.model.ts` / `.interface.ts` files.

### L2-038: BEM naming for component styles
**Traces to:** L1-021

Every component's CSS shall use the BEM convention.

**Acceptance Criteria:**
1. Given a `BalanceCardComponent`, when its styles are inspected, then class names shall follow `block__element--modifier` (e.g. `.balance-card__amount`, `.balance-card__amount--hero`).
2. Given component CSS, when scanned, then no rule shall select by element type or use deep descendant selectors (`*`, `>>>`, `::ng-deep` forbidden outside documented exceptions).

### L2-039: Signals over RxJS, no NgRx
**Traces to:** L1-021

State exposed by services to components shall be `Signal<T>` or `WritableSignal<T>`. RxJS shall be used only where the platform API forces it (e.g. `HttpClient`); results shall be converted to signals at the service boundary using `toSignal()`. NgRx shall not be a dependency.

**Acceptance Criteria:**
1. Given the components library, when its components are inspected, then they shall consume service state via `signal()` reads, not `subscribe()` calls.
2. Given `package.json`, when inspected, then `@ngrx/*` shall not appear as a dependency or devDependency.
3. Given an HTTP call in an api-library service, when written, then the public method shall return `Signal<TResult>` (or a small `Resource`-style object exposing signals), not `Observable<T>`.

### L2-040: Components library is presentation-only
**Traces to:** L1-022

The `components` library shall contain only presentation components. It shall not import `HttpClient`, the `Router`, or any service from the `api` library directly. Data flows in via inputs (signal inputs preferred) and events flow out via outputs.

**Acceptance Criteria:**
1. Given the components library, when its imports are audited, then no file shall import from `@angular/common/http` or `@angular/router`.
2. Given any component in the library, when inspected, then its inputs shall be typed primitives or DTOs from `@tab/api/models` (re-exported types only, no service interfaces), and its outputs shall be `EventEmitter` or signal-output equivalents.

### L2-041: Tab application is a shell
**Traces to:** L1-022

The `tab` application shall contain only routing configuration, route guards, page components that compose presentation components from `@tab/components`, app bootstrap, and global providers wiring `@tab/api` tokens to implementations.

**Acceptance Criteria:**
1. Given the `tab` application source, when audited, then no `.ts` file shall contain UI logic that is not either a page-level composition of components or a route guard / interceptor / provider.
2. Given the app's `main.ts`, when inspected, then it shall call `bootstrapApplication()` with a providers array that includes the `provide…` factory functions exported from `@tab/api`.

## Backend architecture — traces to L1-024 / L1-026 / L1-027

### L2-042: Thin controllers dispatching to MediatR
**Traces to:** L1-024

Every API controller action shall do exactly three things: bind the inbound request to a MediatR `IRequest<TResponse>`, send it via `_mediator.Send(request, ct)`, and return the result wrapped in an appropriate `ActionResult<T>`.

**Acceptance Criteria:**
1. Given any controller action, when inspected, then its body shall be ≤ 5 statements and shall not perform any business logic, validation, or data access directly.
2. Given a controller, when its dependencies are inspected, then it shall depend on `IMediator` only (plus framework-provided services like `ILogger`), never on `DbContext`, repositories, or domain services.

### L2-043: MediatR handlers depend on ITabDbContext
**Traces to:** L1-024

Every MediatR handler that touches data shall depend on `ITabDbContext` (the interface), not on `TabDbContext` (the concrete EF Core class).

**Acceptance Criteria:**
1. Given any class implementing `IRequestHandler<TRequest, TResponse>`, when its constructor is inspected, then it shall not have a parameter of the concrete type `TabDbContext`. It shall accept `ITabDbContext` instead.
2. Given the DI registration, when inspected, then `ITabDbContext` shall be registered as a scoped service resolving to `TabDbContext`.
3. Given `ITabDbContext`, when inspected, then it shall expose every `DbSet<T>` used by the handlers as a property and a `Task<int> SaveChangesAsync(CancellationToken)` method.

### L2-044: No repository layer
**Traces to:** L1-024

The codebase shall not contain a "Repository" pattern abstraction over EF Core. Handlers shall query `DbSet<T>` directly via LINQ.

**Acceptance Criteria:**
1. Given the backend codebase, when class names are scanned, then no class or interface name shall end in `Repository`.
2. Given a handler, when its query logic is inspected, then it shall use `_db.Loans.Where(...).ToListAsync(ct)` style directly.

### L2-045: One public C# type per file
**Traces to:** L1-026

No `.cs` file shall declare more than one public top-level type.

**Acceptance Criteria:**
1. Given any `.cs` file, when scanned, then the count of `public class|record|struct|interface|enum` declarations at the file scope shall be ≤ 1.
2. Given a command and its handler, when authored, then they shall live in two files: `CreateLoanCommand.cs` and `CreateLoanCommandHandler.cs`.
3. Given a response DTO, when authored, then it shall live in its own file named for the response type.

### L2-046: API versioning and Problem Details errors
**Traces to:** L1-027

The API shall be served under `/api/v1/...` and shall use `application/problem+json` for every error response.

**Acceptance Criteria:**
1. Given an unhandled exception in a handler, when caught by the global error filter, then the response shall be `500 application/problem+json` with `type`, `title`, `status`, `detail`, and `traceId` fields, and shall not contain a stack trace.
2. Given a validation failure from FluentValidation, when returned, then the response shall be `400 application/problem+json` with a `errors` dictionary keyed by field name.

### L2-047: SQL Server (Express or LocalDB) with EF Core code-first
**Traces to:** L1-025

The backend shall persist to SQL Server using EF Core code-first migrations.

### L2-048: Entity model
**Traces to:** L1-025

**Acceptance Criteria:**

## Calendar — traces to L1-029

### L2-052: Calendar screen layout & content
**Traces to:** L1-029

The application shall expose a calendar at `/calendar` that renders a single calendar month and places each loan, each payment-in, and each recurring-bill posting (posted or projected) on the date it occurred or is scheduled to occur. Each item shall appear as a typed, colour-coded chip showing its amount; the three types — `loan`, `payment`, `bill` — shall be visually distinct, and projected (not-yet-paid) bill postings shall be visually distinguished from posted bill postings (e.g. dashed border / lower opacity).

**Acceptance Criteria:**
1. Given the user navigates to `/calendar`, when the page first renders, then the displayed month shall default to the current month and today's date cell shall carry a visible "today" indicator.
2. Given a date has one loan entry, one payment-in, and one posted bill split, when the date cell renders, then three chips shall appear in that cell, each labelled with its type and amount, ordered loans → bill splits → payments-in.
3. Given a recurring bill is defined with split percentage 50% and expected amount $168, and the next two due dates fall in months the user has not yet marked paid, when those months are viewed, then two projected `bill` chips of $84.00 each shall appear on those due dates, each rendered with the projected styling and a tooltip / aria-label of "Projected — not yet posted".
4. Given a posted bill split already exists for a period, when that month is viewed, then exactly one chip shall be rendered for that bill on that date (no duplicate projected chip overlaid on the same posted entry).
5. Given a date cell contains more chips than fit in the available height, when rendered, then the cell shall truncate to the first three chips and show a "+N more" affordance that, on activation, reveals the full list for that day.
6. Given the user has zero entries and zero recurring bills, when the page renders, then the calendar grid shall still render with no chips, and an empty-state banner above the grid shall read "No activity yet — add a loan, log a bill, or define a recurring bill to see it here".
7. Given the page is loaded by an authenticated user, when the bound payload is inspected, then it shall contain only entries owned by that user (verified server-side per L2-005).

### L2-053: Calendar navigation and date range
**Traces to:** L1-029

The calendar shall let the user move between months, jump to the current month, and switch to an agenda (chronological list) view; the visible range shall always be reflected in the URL so the view is shareable and back-button safe.

**Acceptance Criteria:**
1. Given the user is viewing May 2026, when they activate "Previous month", then the URL shall update to `/calendar?month=2026-04` and April 2026 shall render.
2. Given the user is viewing any month other than the current one, when they activate "Today", then the URL shall update to `/calendar?month={current YYYY-MM}` and the current month shall render with today highlighted.
3. Given the user loads `/calendar?month=2027-01` directly, when the page renders, then January 2027 shall be shown with all projected bill postings for that month visible.
4. Given the user toggles to agenda view, when the view switches, then entries shall be listed chronologically with date headers and the same chip styling, and the URL shall update to `/calendar?month=YYYY-MM&view=agenda`.
5. Given the user presses the browser back button after navigating across three months, when each prior state is restored, then the calendar shall render the month that was previously shown.

### L2-054: Calendar entry interaction
**Traces to:** L1-029

The user shall be able to activate any chip to view or edit the underlying entry.

**Acceptance Criteria:**
1. Given a chip representing a loan, when the user activates it, then the loan editor shall open pre-populated (per L2-009).
2. Given a chip representing a payment-in, when the user activates it, then the payment editor shall open pre-populated with the payment's values.
3. Given a chip representing a posted bill split, when the user activates it, then a detail sheet shall open showing the bill name, the period, the amount, and a link to the bill definition.
4. Given a chip representing a projected (not-yet-posted) bill, when the user activates it, then the same "Mark paid in full" / "Log this month" actions available on `/bills` (per L2-013) shall be presented, scoped to that bill and that period.
5. Given a chip is focused via keyboard, when the user presses Enter or Space, then the same activation behaviour shall occur as for a pointer activation.

### L2-055: Calendar API composite endpoint
**Traces to:** L1-027

The backend shall expose `GET /api/v1/calendar?from=YYYY-MM-DD&to=YYYY-MM-DD` that returns, in a single response, every posted ledger entry in the range plus every projected bill posting in the range computed from active recurring-bill definitions.

**Acceptance Criteria:**
1. Given a valid `from`/`to` range covering one calendar month, when the endpoint is invoked, then the response shall be `200` with a body containing two collections: `entries` (every posted loan, payment-in, and bill-split in the range, each with `id`, `date`, `type`, `amount`, `description`) and `projections` (every future-or-current bill-due date in the range that has no matching posted bill-split, each with `billId`, `date`, `expectedAmount`, `counterpartyShare`, `billName`).
2. Given a range spans more than 366 days, when the endpoint is invoked, then the response shall be `400 application/problem+json` with `detail` = "Range exceeds maximum of 366 days".
3. Given the authenticated user has no recurring bills and no ledger entries in the range, when invoked, then the response shall be `200` with `{ entries: [], projections: [] }`.
4. Given a bill posting already exists for a given bill and period inside the requested range, when the response is built, then no projection shall be emitted for that bill on that period (no double-counting with L2-052 #4).
5. Given the request omits `from` or `to`, when received, then the response shall be `400 application/problem+json` flagging the missing field.
6. Given a request from user A includes a bill `id` belonging to user B, when processed, then the response shall exclude that bill entirely and shall not error (per L2-005 ledger isolation).

### L2-056: Calendar in primary navigation
**Traces to:** L1-029

The Calendar screen shall be reachable as a top-level destination in the primary navigation alongside Dashboard and Loans, on every viewport.

**Acceptance Criteria:**
1. Given the viewport is XS/S/M, when any page renders, then the bottom navigation bar shall include a "Calendar" tab as a top-level destination between "Dashboard" and "Loans" with an appropriate icon and ≥ 44 × 44 px tap target.
2. Given the viewport is L/XL, when any page renders, then the left sidebar shall include a "Calendar" link in the same primary group as "Dashboard" and "Loans".
3. Given the user is on `/calendar`, when the navigation renders, then the Calendar tab/link shall carry the active state and shall be announced as `aria-current="page"` by screen readers.

### L2-057: Calendar responsive layout
**Traces to:** L1-018

The calendar shall present a month grid at M and wider, and shall collapse to a date-grouped agenda list at XS/S widths where a 7-column grid is not legible.

**Acceptance Criteria:**
1. Given the viewport is XS or S, when `/calendar` renders, then the default view shall be the agenda list (chronological entries with sticky date headers), and a toggle to month grid shall be available but secondary.
2. Given the viewport is M, L, or XL, when `/calendar` renders, then the default view shall be a 7-column month grid spanning the full content width with date cells of equal width and a minimum cell height of 96 px at M, 120 px at L, 140 px at XL.
3. Given the viewport is any width, when the page renders, then no element shall cause horizontal scrolling and chip text shall truncate with an ellipsis rather than wrap onto a third line.

### L2-058: Calendar performance budget
**Traces to:** L1-017

The calendar payload for a single month shall return within the per-endpoint budget defined in L2-026, even on a 10 000-entry ledger.

**Acceptance Criteria:**
1. Given a seeded test database with 10 000 ledger entries and 12 active recurring bills for one user, when `GET /api/v1/calendar?from=2026-05-01&to=2026-05-31` is invoked 100 times sequentially, then the 95th-percentile response time shall be ≤ 300 ms on CI hardware.
2. Given the same query, when its SQL is captured, then it shall execute in ≤ 2 round-trips (one for posted entries, one for active bill definitions) with no N+1 patterns over recurring bills.
